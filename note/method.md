### LoRR2026 方法设计笔记

> 文档定位：可行方案、分赛道改造方向、迭代优先级与策略选择。

这份文档不是直接给出一个“唯一正确算法”，而是把当前题目的**本质约束**、**性能瓶颈**和**几条真正可落地的方法路线**整理出来，方便后续做方案选择。

文档结论基于当前仓库中的系统流程、公开样例规模、时间约束和已有 `alg.md` / `tech.md` / `paper.md` 的梳理。

### 1. 这道题的本质是什么

如果只看表面，它像是“多机器人路径规划”；但从系统实际行为看，它更接近：

- **在线任务分配**：任务不是一次性给完，而是持续补充；
- **终身多智能体路径规划**：机器人完成一个任务后还要继续接新任务；
- **受扰动执行**：运行时存在 `delay`，计划和真实执行会发生偏差；
- **双时间尺度控制**：慢环重规划，快环执行放行；
- **有限时间预算下最大化吞吐**：核心目标不是静态最短路，而是在固定仿真时长内尽量完成更多任务。

从输出字段和主流程看，最核心指标仍然是：

- **主目标**：`numTaskFinished` 尽量高；
- **约束指标**：`numPlannerErrors`、`numScheduleErrors`、`numEntryTimeouts` 尽量低；
- **过程指标**：`plannerTimes` 稳定，`actualPaths` 不要和 `plannerPaths` 严重背离。

所以这题真正要优化的是：

\[
\text{吞吐量} = \text{任务完成数} - \text{错误/超时/拥堵带来的损失}
\]

而不是单个 agent 的路径长度最优。

### 2. 当前系统带来的关键约束

### 2.1 规划不是每个 tick 都发生

当前系统是明显的**双环结构**：

- **慢环**：周期性调用 `Entry::compute(...)`，先做任务分配，再生成多步动作；
- **快环**：每个 tick 调 `Executor::next_command(...)`，决定当前哪些 agent `GO`、哪些 `STOP`。

这意味着：

- 规划器必须输出**一段时间内都还能用**的多步计划；
- 执行器必须能在计划逐渐失真时继续兜底；
- 如果慢环太慢，系统不会等你，而是继续跑旧计划或者原地等。

### 2.2 任务是持续在线补充的

任务池不是按固定时刻整体刷新，而是始终补到 `numTasksReveal * teamSize` 左右。

因此调度问题不是“给定全集求一次最优匹配”，而是：

- 不断有 agent 空闲出来；
- 不断有任务被补进池子；
- 当前一次分配会影响未来几轮拥堵和可用资源。

### 2.3 任务具有串行 errand 结构

一个 task 可能包含多个站点，而且必须按顺序完成。

这带来两个后果：

- 调度评分不能只看“到第一个点多远”，还要考虑整条 task 链的成本；
- 一旦任务已经被“打开”（第一个 errand 完成），重分配自由度会下降，所以**首次分配质量很重要**。

### 2.4 执行层和真实安全模型不能忽略

当前不是纯离散顶点/边冲突模型，而是：

- 动作跨多个 tick 执行；
- 运行时可能被 delay 打断；
- 最终是否能动，还要经过 executor 和 action model 的安全检查。

这意味着：

- 即使 planner 给出网格层面无冲突的计划，真实执行中仍可能被迫 `STOP` 或 `WAIT`；
- 只优化静态路径质量，不优化执行鲁棒性，成绩往往上不去。

### 2.5 公开样例已经说明了规模要求不低

从仓库公开样例看：

- **地图尺度**：从 `32 x 32` 到 `481 x 530`；
- **机器人数量**：从 `100` 到 `5000`；
- **任务池倍率**：公开样例主要在 `1` 或 `1.5`；
- **delay 配置**：公开样例都带 `delayConfig`。

这说明真正可行的方法必须满足两点：

- **能扩展到大规模**；
- **能容忍执行扰动**。

任何需要频繁全局最优求解、或者每 tick 做复杂全局搜索的方法，落地风险都很高。

### 3. 我认为最值得抓的三个核心矛盾

### 3.1 近任务不一定是好任务

默认调度偏向距离近、估计 makespan 小的任务，但这会忽略：

- 任务目的地是否在热点区域；
- 该 agent 是否正处在拥堵走廊；
- 把这个 agent 派走后，本区域是否会失去服务能力；
- 该任务后续链条是否会穿越多个瓶颈。

所以调度的核心矛盾是：

- **局部最近** vs **系统吞吐最高**。

### 3.2 最短路不一定是最高吞吐路径

在 Lifelong MAPF / 仓储场景里，大家都走最短路，往往会把瓶颈压爆。

规划层真正需要优化的是：

- 到达目标要快；
- 但更重要的是**不要制造长时间对向流、交叉流和死锁链**。

所以规划的核心矛盾是：

- **单体路径短** vs **整体交通顺**。

### 3.3 保守执行不一定更安全

如果 executor 过于保守，会导致大量 `STOP`，吞吐量显著下降；
如果过于激进，又可能放大 delay 带来的级联阻塞。

因此执行层的核心矛盾是：

- **局部安全** vs **持续放行能力**。

### 4. 可行方案总览

下面把我认为真正值得尝试的方法按复杂度和风险分成四类。

| 路线 | 核心想法 | 适合赛道 | 开发成本 | 预期收益 |
| --- | --- | --- | --- | --- |
| 方案 A | 做更聪明的调度评分 | Scheduling / Combined | 低 | 稳定提升，见效快 |
| 方案 B | 做拥堵感知的滚动规划 | Combined | 中 | 中高，尤其在中大图 |
| 方案 C | 做 delay-aware 执行控制 | Execution / Combined | 中 | 对鲁棒性提升明显 |
| 方案 D | 做分层协同系统 | Combined | 高 | 上限最高，但调参与实现都更难 |

### 5. 方案 A：从调度入手的低风险提升

这是我最推荐的第一步，因为它通常：

- 改动小；
- 不容易破坏正确性；
- 对 `numTaskFinished` 往往有直接帮助。

### 5.1 拥堵感知的任务打分

不要只按“agent 到任务首站 + task 内部距离”排序，可以改成综合评分：

\[
score(a, t)=d_{start}(a,t)+\lambda_1 d_{task}(t)+\lambda_2 c_{hot}(t)+\lambda_3 c_{region}(a,t)+\lambda_4 c_{queue}(t)
\]

其中可以包含：

- **`d_start`**：agent 到任务第一个 errand 的距离；
- **`d_task`**：任务内部多站点的累计距离；
- **`c_hot`**：任务目标是否处在热点/瓶颈区域；
- **`c_region`**：把该 agent 调走后，本区域会不会变空；
- **`c_queue`**：目标附近是否已有很多 agent 的目标重叠。

这类方法的优点是：

- 不需要改系统框架；
- 很符合这题“吞吐优先”的目标；
- 即使 planner 保持 baseline，也常能提升实际效果。

### 5.2 区域负载均衡

把地图粗分成若干区域，维护：

- 区域内空闲 agent 数；
- 区域内待服务 task 数；
- 区域间迁移成本；
- 区域热点强度。

调度时加入一个简单原则：

- **不要为了当前一个近任务，把某个区域服务能力抽空**。

可行做法：

- 给每个区域保留最少服务容量；
- 当某区域即将“空仓”时，提高本区域任务优先级；
- 对跨区域任务加迁移惩罚。

这对大仓图和长走廊地图尤其有用。

### 5.3 预留 / 停车位意识

对空闲 agent，不一定立刻给它最便宜的新任务；有时更好的做法是：

- 让它停在更适合作为下一轮出发点的位置；
- 或者优先靠近高需求区域边界，而不是钻进拥堵核心。

可以理解成一种轻量 rebalancing：

- **不是所有 idle 都要马上接任务**；
- 有时候“站位正确”比“马上接一个次优任务”更值钱。

### 5.4 这条路线最适合谁

- **Scheduling Track**：最值得优先做；
- **Combined Track**：也应该先做，因为它成本最低、收益最稳；
- **Execution Track**：不适用。

### 6. 方案 B：拥堵感知的滚动规划

如果你能改 planner，那么最值得做的不是追求单次全局最优，而是做一个**可重复稳定返回的 rolling horizon planner**。

### 6.1 自适应 horizon

当前系统本身就是慢环重规划，所以一个自然思路是：

- 密度高 / delay 强时，缩短 horizon，优先保证快速返回和稳定执行；
- 密度低 / 路网开阔时，拉长 horizon，多做前瞻避堵。

可用的自适应信号包括：

- 当前活跃任务数；
- 目标重叠度；
- 过去几轮 `plannerTimes`；
- `actualPaths` 与 `plannerPaths` 的偏离程度；
- executor 中 `STOP` 比例。

这条思路的核心不是“固定长规划最好”，而是：

- **规划长度要和可执行性、可返回性匹配**。

### 6.2 瓶颈规避与单向流偏好

对于长走廊、窄门、汇聚路口，可以显式引入：

- 单向偏好；
- 进入惩罚；
- 高峰期绕行；
- 相反流量抑制。

这和 `paper.md` 里 traffic flow / guidance graph 的思路是一致的。

即使不实现复杂的全局 flow 优化，也可以做轻量版本：

- 统计最近若干轮哪些边/格子拥堵最严重；
- 在路径代价中加热度项；
- 对“对向穿越瓶颈”的行为施加额外惩罚。

规划代价可以写成：

\[
cost(path)=length + \alpha \cdot congestion + \beta \cdot reverseFlow + \gamma \cdot hotspotCross
\]

### 6.3 只重算真正需要重算的 agent

因为系统是在线滚动的，没必要每轮都把所有 agent 从头规划一遍。

更实际的做法是只重点更新：

- 新接任务的 agent；
- 目标发生变化的 agent；
- 处在热点区且近期反复被阻塞的 agent；
- 当前路径已经明显失效的 agent。

其余 agent：

- 继续沿用原有 guide path / staged plan；
- 或只做 very light repair。

这样更符合这题的时间限制，也更适合大规模实例。

### 6.4 给 planner 留“可执行余量”

规划时不要把每条路都压到极限。

一个很有用的原则是：

- **宁可路径略长，也要给 executor 留局部让路空间**。

具体表现为：

- 少走狭窄汇聚点；
- 减少 head-on；
- 避免多个 agent 同时瞄准同一热点格；
- 给空闲 agent 指定停车区，而不是停在主干道上。

### 6.5 这条路线最适合谁

- **Combined Track**：非常值得；
- **Scheduling Track**：只能间接受益；
- **Execution Track**：不能直接做。

### 7. 方案 C：delay-aware 的执行器增强

如果你能改 `Executor`，这通常是另一条很有潜力的路线，因为当前比赛显式存在 delay。

### 7.1 从“只看下一步”升级到“看短窗口依赖” 

默认 executor 更偏向当前一步的局部可通行判断。可加强为：

- 看接下来 `k` 个 staged action；
- 判断某个 agent 一旦放行，是否会把自己送进短期死胡同；
- 区分“短等一下更好”和“现在就必须抢先通过”的情形。

这类短窗口依赖图方法通常比纯一步贪心更稳。

### 7.2 阻塞链拆解与公平性

在高密度图里，常见问题不是单点冲突，而是长链阻塞。

可以给 executor 增加：

- **等待年龄**：等太久的 agent 提高放行优先级；
- **阻塞贡献度**：谁卡住更多人，谁优先被疏散；
- **死锁破坏规则**：出现循环等待时，强制选一个方向破局；
- **热点保护**：避免把 agent 停在关键通道中央。

这类规则往往不复杂，但对吞吐改善很实用。

### 7.3 delay 感知的计划消费

由于 delay 会让 staged plan 逐渐过期，executor 可以更主动地处理：

- 对已明显失配的后续动作做截断；
- 对局部顺序做轻量重排；
- 必要时把 agent 引导到一个安全等待位，而不是一直卡在主路上。

如果是 Combined Track，还可以让 planner 在 `Plan` 中附带额外元信息，例如：

- 路径优先级；
- 局部避让等级；
- 推荐等待点；
- 瓶颈通行顺序。

然后由 executor 解释这些信息。

### 7.4 这条路线最适合谁

- **Execution Track**：最值得优先做；
- **Combined Track**：也非常推荐；
- **Scheduling Track**：不能直接做。

### 8. 方案 D：分层协同的高上限方案

如果你打 Combined Track，真正上限高的方案通常不是单点改进，而是三层联动：

- **调度层**：决定谁去哪里；
- **规划层**：决定全局怎么绕堵；
- **执行层**：决定局部怎么稳住。

### 8.1 一个很实用的分层结构

我比较推荐的整体结构是：

- **上层区域调度**：控制区域负载与任务分配；
- **中层滚动路径规划**：给活跃 agent 提供 congestion-aware 多步 guide path；
- **下层执行控制**：根据 delay 和局部阻塞做实时放行。

这种结构的优点是：

- 各层职责清楚；
- 可逐层调试；
- 更容易针对大规模地图做近似，而不是指望单个算法包打天下。

### 8.2 典型协同信号

三层之间可以传的信号包括：

- **调度 -> 规划**：区域优先级、热点惩罚、推荐停车区；
- **规划 -> 执行**：瓶颈顺序、优先通行标签、等待点；
- **执行 -> 调度/规划**：高阻塞区域、经常失配的 agent、真实 `STOP` 热点。

真正有用的，不是把所有状态都传来传去，而是传这些**能改变决策的摘要量**。

### 8.3 为什么这条路线风险更高

因为它会引入：

- 更多参数；
- 更多模块耦合；
- 更复杂的实验设计；
- 更难定位性能波动来源。

所以这条路线适合作为第二阶段，而不是一上来就全做。

### 9. 按赛道给出更具体的建议

### 9.1 Scheduling Track

最推荐的顺序：

1. **拥堵感知评分**
2. **区域负载均衡**
3. **idle 站位 / rebalancing**
4. **任务链长度和热点代价建模**

原因很简单：

- 你不能改 planner / executor；
- 但默认 planner 是 anytime，调度越快且越合理，整体越容易涨分；
- 错误调度代价很大，因为 opened task 的重分配空间有限。

### 9.2 Execution Track

最推荐的顺序：

1. **短窗口依赖图放行**
2. **等待年龄 + 死锁破坏机制**
3. **delay-aware staged action 截断/修复**
4. **热点通道保护**

这条赛道里，别把 executor 写成“另一个 planner”。

更现实的目标是：

- 每 tick 足够快；
- 比默认规则更少无谓 `STOP`；
- 在 delay 下不至于级联崩坏。

### 9.3 Combined Track

最推荐的顺序：

1. **先做更好的调度**；
2. **再做轻量拥堵感知规划**；
3. **再做执行器鲁棒增强**；
4. **最后再考虑自定义 `Plan` 元信息或学习方法**。

原因是：

- 先把系统吞吐的下限抬起来；
- 再逐步减少拥堵与执行偏差；
- 这样实验更可控，不容易一次引入太多变量。

### 10. 我最推荐的三条实操路线

如果你现在就要决定怎么开工，我建议只在下面三条里选一条起步。

### 10.1 路线一：稳健提分路线

适合：**想先稳定超过 baseline**。

做法：

- 调度里加入拥堵与区域负载项；
- 规划保持滚动多步，但减少热点穿越；
- executor 只加最小必要的公平性和死锁破坏规则。

特点：

- 风险低；
- 容易 ablation；
- 对公开样例通常就会有效。

### 10.2 路线二：执行鲁棒路线

适合：**Execution Track 或非常关注 delay**。

做法：

- 强化 `next_command(...)`；
- 维护短窗口依赖；
- 避免无谓停滞；
- 对已失真的 staged actions 做修复。

特点：

- 对 delay-heavy 场景更可能有效；
- 需要大量看 `actualPaths` 与 `plannerPaths` 的偏差。

### 10.3 路线三：交通组织路线

适合：**Combined Track，想追求更高上限**。

做法：

- 显式识别地图瓶颈、主干道、停车带、服务热点；
- 用区域调度 + 单向偏好 + 热点惩罚组织整体交通；
- 用滚动规划和执行器共同维护这种交通秩序。

特点：

- 在大图和高机器人密度下潜力最大；
- 但需要较多地图分析和调参。

### 11. 哪些思路我不建议作为第一步

下面这些方向并不是不能做，而是我认为**不适合作为第一步**：

- **重型全局最优 MAPF**：在大规模、在线、受 delay 扰动场景下，成本太高；
- **每轮都重算所有 agent 的精细路径**：时间不稳，且很多工作会被 delay 抵消；
- **一上来就上学习方法**：训练、部署、调参链路都更长，不适合先验证系统瓶颈；
- **只盯着 planner，不管 executor**：现实执行偏差会吞掉一大块理论收益；
- **只做局部最近任务分配**：在热点区容易把拥堵越做越严重。

### 12. 实验与验证建议

不管选哪条路线，我都建议固定看下面这些指标。

### 12.1 必看指标

- **`numTaskFinished`**：总目标；
- **`numPlannerErrors` / `numScheduleErrors`**：正确性底线；
- **`numEntryTimeouts`**：是否因为方法太重导致慢环掉频；
- **`plannerTimes`**：时间是否稳定，是否经常贴边；
- **`actualPaths` vs `plannerPaths`**：执行偏差到底大不大。

### 12.2 我建议额外自己统计的指标

- **热点格访问频率**；
- **各区域空闲 agent 数**；
- **agent 平均等待比例**；
- **executor 中 `STOP` 占比**；
- **任务从 reveal 到完成的平均时长**；
- **长链阻塞出现次数**。

这些指标能帮助你判断问题到底出在：

- 调度差；
- 规划差；
- 还是执行层吞吐不够。

### 12.3 推荐 ablation 顺序

最稳的实验顺序一般是：

1. 只改调度评分；
2. 再加区域项；
3. 再加规划层热点惩罚；
4. 再加 executor 公平性/死锁破坏；
5. 最后才做更复杂的联动。

这样你比较容易知道是哪一步真正带来了收益。

### 13. 实现层面的现实提醒

这里有一个很实际的约束：

- 评测时很多 start-kit 文件会被覆盖；
- 某些 baseline 实现文件是受保护的；
- 不同 track 下可修改入口不同。

因此真正可行的实现方式应该是：

- 在**允许修改的接口层**接入你自己的实现；
- 把算法主体写到新的自定义文件中；
- 不要把希望寄托在修改受保护 baseline 文件上。

换句话说，方法设计时就要考虑：

- **这个思路最终要挂到哪个可修改 API 上？**
- **是 `TaskScheduler`、`MAPFPlanner`、`Executor`，还是 Combined 下的 `Entry`？**

### 14. 按三个核心模块进一步深化方案

这一节专门回应三个更落地的问题：

1. **delay 发生后，每个 tick 到底让谁动**；
2. **任务如何分配，尤其是初始化与动态过程如何区别处理**；
3. **任务确定后，路径如何做全局或增量规划**。

核心思路是：

- **任务分配**解决“谁该去哪里”；
- **路径规划**解决“打算怎么走”；
- **执行控制**解决“这一 tick 现实中能不能走”。

这三层不应混为一个算法，但必须协同设计。

### 14.1 delay 发生后，每个 tick 如何决定 agent 是否移动

### 14.1.1 先把问题说清楚

在这个系统里，planner 不是每 tick 都重算，而 executor 是每 tick 都要给出 `GO/STOP`。因此一旦出现 delay，真正困难的不是“重新算一条理论最优路”，而是：

- 当前 agent 有没有必要继续推进；
- 它推进后会不会把自己或别人送入更差的阻塞链；
- 是不是应该让某个 bottleneck 上的 agent 优先通过；
- 已经延迟太久的 agent，是否应该被提升优先级。

也就是说，**delay 下的每 tick 决策，本质上是一个局部时序调度问题**，而不只是静态碰撞检查。

### 14.1.2 论文给出的启发

这部分最值得参考三条线：

- **`Multi-Agent Path Finding with Delay Probabilities`**
  - 启发点：把计划执行看成一个**时序依赖问题**；谁先访问某位置，谁后访问某位置，最好在执行时继续被尊重。
  - 对当前系统最直接的启发，就是 **TPG / dependency graph** 思路。

- **`Robust Multi-Agent Pickup and Delivery with Delays`**
  - 启发点：delay 不只是噪声，而是执行层必须主动建模的现实约束。
  - 重点不是频繁全局重规划，而是**让执行策略本身更鲁棒**，尽量减少级联停滞。

- **`Priority Inheritance with Backtracking for Iterative Multi-agent Path Finding`**
  - 启发点：当局部冲突出现时，不能只看当前 agent，而要允许**优先级继承和链式让路**。
  - 这很适合用来设计 delay 下的局部放行逻辑。

### 14.1.3 三种可选执行策略

| 方案 | 核心机制 | 优点 | 缺点 | 适用性 |
| --- | --- | --- | --- | --- |
| 严格 TPG 放行 | 严格按计划访问顺序放行 | 安全、简单、稳定 | 容易过于保守，`STOP` 多 | 低风险 baseline |
| 短窗口风险感知 TPG | 看未来 `k` 步依赖，结合风险分数放行 | 吞吐更高，能处理短链阻塞 | 实现稍复杂 | **最推荐** |
| 激进局部抢占 | 谁现在能走就尽量走 | 局部吞吐高 | delay 下容易制造级联阻塞 | 不建议直接采用 |

### 14.1.4 我更推荐的方案：短窗口风险感知 TPG

核心思想是：**保留 TPG 的顺序约束，但不把它做成“只要前面有人就一律停”**。

具体可以这样做：

#### A. 为每个 agent 维护短窗口依赖

不是只看下一步，而是看接下来 `k` 个 staged actions，通常可取：

- `k = 2 ~ 4`：适合大规模高频执行；
- `k > 4`：收益可能变小，但执行器耗时会明显上升。

对每个 agent，提取：

- 下一目标格；
- 下一关键 bottleneck 是否会被占；
- 它后面是否还有链式跟随者；
- 当前计划与真实状态的偏离程度。

#### B. 给每个候选 `GO` 计算一个放行优先级

可以定义：

\[
P_i = w_1 \cdot age_i + w_2 \cdot bottleneck_i + w_3 \cdot critical_i - w_4 \cdot risk_i - w_5 \cdot stale_i
\]

其中：

- **`age_i`**：这个 agent 已连续等待了多久；
- **`bottleneck_i`**：它是不是正处在瓶颈入口 / 出口；
- **`critical_i`**：它后面是否拖着较长阻塞链；
- **`risk_i`**：它若前进，是否会把冲突扩散到更多邻居；
- **`stale_i`**：它当前 staged path 是否已经明显失真。

可以再把风险项拆成：

\[
risk_i = \alpha \cdot chain_i + \beta \cdot hotspot_i + \gamma \cdot mismatch_i
\]

#### C. 每个 tick 的决策流程

推荐流程：

1. **若 agent 正在 delay 中**：直接 `STOP`；
2. **若没有 staged action**：`STOP`；
3. **若下一动作不是 `FW`**：通常允许 `GO`，因为它主要是在原地转向；
4. **若下一动作是 `FW`**：
   - 检查目标格是否可进入；
   - 检查目标格相关的依赖顺序；
   - 计算 `P_i`；
5. 按 `P_i` 从高到低选择一个**可行的放行集合**；
6. 如果出现环形等待：
   - 选出一个 leader 强制先行；
   - 其余 agent `STOP`；
7. 如果某 agent 连续等待超过阈值：
   - 临时提高其优先级；
   - 或标记为下一轮 planner 的重点修复对象。

### 14.1.5 为什么这个方案比“只看当前一步”更好

因为 delay 的危险不在于“当前格子能不能走”，而在于：

- 走了之后是否堵死后面两三步；
- 是否把 bottleneck 占住太久；
- 是否让其他 agent 的计划整体报废。

短窗口风险感知的价值就在这里：

- 仍然足够轻量；
- 但能显式区分“该让一步”和“该抢一次 bottleneck”；
- 更适合当前这种 **planner 慢环 + executor 快环** 的结构。

### 14.1.6 最终建议

如果只做执行层，我建议按下面顺序实现：

1. **先保留 TPG 主体**；
2. **加入等待年龄与 bottleneck 优先级**；
3. **再加入短窗口依赖与 stale-plan 检测**；
4. **最后才考虑学习型或更复杂的局部策略**。

一句话概括：

**delay 下的 executor 不应该只是“碰撞检查器”，而应该是一个轻量的局部时序调度器。**

### 14.2 如何进行任务分配

任务分配我建议明确分成两个阶段来想：

- **初始化分配**：系统刚启动时，所有 agent 基本都空闲；
- **动态分配**：运行过程中，只有部分 agent 新空闲、部分新任务 reveal。

这两个阶段不应该用完全同一种策略。

### 14.2.1 初始化分配：更适合做偏全局的布局

初始化阶段有两个特征：

- 可用 agent 很多；
- 当前这次分配会决定系统早期交通格局。

如果这一步只是“谁最近谁去”，常见后果是：

- 大量 agent 同时扑向局部热点；
- 某些区域被抽空；
- 很快在主通道形成对向流。

因此初始化更适合考虑**全局平衡**。

### 14.2.2 初始化阶段的三种方案

| 方案 | 核心思想 | 优点 | 缺点 | 评价 |
| --- | --- | --- | --- | --- |
| 最近距离贪心 | 最近 agent 接最近 task | 快、简单 | 容易早期拥堵 | 只适合作 baseline |
| 全局匹配 | 在 agent-task 二部图上做加权匹配 | 早期布局较好 | 大规模时较重 | 中等可行 |
| 区域引导 + 局部匹配 | 先平衡区域，再在区域内匹配任务 | 兼顾全局与规模 | 需要区域建模 | **最推荐** |

### 14.2.3 我更推荐的初始化方案：区域引导 + 局部匹配

步骤可以是：

#### A. 先做区域级资源布局

把地图切成若干区域，区域划分方式可以是：

- 均匀网格切块；
- 走廊 / 功能区切块；
- 基于 bottleneck 分割。

对每个区域统计：

- 当前 reveal task 数；
- 区域热点强度；
- 与周边区域的交通代价；
- 该区域至少需要保留多少 agent。

然后先做一个粗粒度目标：

- 哪些 agent 应该留在本区域；
- 哪些 agent 可以跨区支援；
- 哪些区域需要预留机动容量。

#### B. 再做区域内局部匹配

对每个 agent 和 task 定义成本：

\[
C_{a,t}^{init} = d_{start}(a,t) + \lambda_1 d_{chain}(t) + \lambda_2 congestion(t) + \lambda_3 imbalance(region(t))
\]

其中：

- **`d_start`**：到任务首个 errand 的距离；
- **`d_chain`**：任务内部链长度；
- **`congestion`**：目标区域是否是热点；
- **`imbalance`**：该区域是否已经被分配过多 agent。

然后在每个区域内做：

- 小规模 Hungarian / min-cost matching；或
- 剪枝后的贪心匹配。

### 14.2.4 为什么不建议初始化只做最近距离

因为初始化阶段一旦形成坏的交通模式，后续几轮会持续付出代价：

- executor 会更频繁 `STOP`；
- planner 会一直在修拥堵；
- 已打开的任务不容易重分配。

所以初始化分配的价值，不是“第一批任务看起来更近”，而是：

- **把系统带入一个更好组织的初始交通状态**。

### 14.2.5 动态任务分配：更适合事件驱动 + 增量更新

运行过程中并不是每轮都要全局重做一次 assignment。

更合理的触发条件通常是：

- 有新的 `new_freeagents`；
- 有新的 `new_tasks`；
- 某区域拥堵过高，需要主动 rebalancing；
- 某些 agent 长时间卡住，需要换目标或临时预留。

也就是说，动态任务分配更适合做成：

- **事件驱动**；
- **局部子问题求解**；
- **尽量保持 assignment 稳定**。

### 14.2.6 动态阶段的三种方案

| 方案 | 核心思想 | 优点 | 缺点 | 评价 |
| --- | --- | --- | --- | --- |
| 每轮全局重分配 | 所有人和所有任务重新匹配 | 形式统一 | 太重，扰动大 | 不推荐 |
| 新增 agent / task 局部贪心 | 只给新空闲 agent 找最近任务 | 快 | 容易短视 | 可作 fallback |
| 事件驱动增量匹配 | 只在受影响子集上做带切换惩罚的匹配 | 稳定、可扩展 | 需要候选筛选 | **最推荐** |

### 14.2.7 我更推荐的动态方案：事件驱动增量匹配

关键原则有四条：

#### A. 已打开任务尽量冻结

一旦任务已经开始执行，原则上不再轻易更换 agent。

这与 MAPD 论文里的思想一致：

- 开始服务后的任务具有更强的路径依赖；
- 频繁改派会让 planner 与 executor 都承受额外代价。

#### B. 只对候选子集做分配

不要让每个空闲 agent 看所有任务，可以做两层剪枝：

- 每个空闲 agent 只保留 top-`k` 候选任务；
- 每个新任务只保留 top-`k` 候选 agent。

这样可以把大规模问题缩成小规模局部匹配。

#### C. 加入切换惩罚与区域惩罚

动态阶段的代价函数建议写成：

\[
C_{a,t}^{dyn} = d_{start}(a,t) + \lambda_1 d_{chain}(t) + \lambda_2 hotspot(t) + \lambda_3 imbalance(t) + \lambda_4 switch(a,t)
\]

其中：

- **`switch(a,t)`**：如果这次分配会让 agent 偏离原来的良好站位，就加罚；
- 也可以对“离开当前紧缺区域”加额外惩罚。

#### D. 保留一小部分机动容量

不要把所有空闲 agent 都立即派满。

对于高需求区域，建议：

- 保留少量 idle / parking agents；
- 让它们停在不堵主路的位置；
- 当附近出现新任务时快速接手。

这其实是从论文中的 rebalancing / token passing 思路借来的一个工程化版本。

### 14.2.8 最终建议

如果只改调度，我建议用一个**二阶段混合方案**：

- **初始化**：区域引导 + 局部匹配；
- **动态过程**：事件驱动增量匹配 + 切换惩罚 + 局部 rebalancing。

这样做比“纯最短距离分配”更适合这道题，因为它优化的是：

- 任务完成节奏；
- 区域服务能力；
- 未来拥堵，而不是单次出发距离。

### 14.3 确定任务后如何进行全局或增量路径规划

路径规划这部分，我建议不要只问“用 beam search 还是优先级”。更准确的问法应该是：

- 哪些 agent 需要全局重算；
- 哪些 agent 只需要局部修复；
- 冲突解决是做成**优先级搜索**，还是做成**宽度受限的候选分支搜索**。

### 14.3.1 论文给出的三条主要路线

- **RHCR / rolling horizon**
  - 启发点：终身 MAPF 不适合每次把整场未来都算完，更适合按窗口重规划。

- **PIBT / prioritized planning 一线**
  - 启发点：高频场景里，优先级方法往往比全局最优方法更可扩展。

- **traffic flow / highway / guidance graph**
  - 启发点：在大规模拥堵环境下，必须给路径提供“交通组织结构”，否则最短路会把系统拖死。

因此更好的问题不是二选一，而是：

- **全局层面**：用 highway / guidance 组织大方向；
- **局部冲突层面**：用优先级或小规模 beam 做冲突消解；
- **动态更新层面**：只重算受影响子集。

### 14.3.2 三种候选规划架构

| 方案 | 核心机制 | 优点 | 缺点 | 评价 |
| --- | --- | --- | --- | --- |
| 全局重规划 | 每轮为全部 active agents 重算 | 理念统一 | 大规模时过重 | 不推荐 |
| 优先级增量规划 | 先定优先级，再按 reservation / time-space 搜索逐个规划 | 稳定、可解释、可扩展 | 顺序敏感 | **最推荐** |
| beam-search 局部分支规划 | 对热点冲突子集保留多个候选分支 | 适合局部复杂冲突 | 启发式敏感，宽度难调 | 适合作补充模块 |

### 14.3.3 我更推荐的主路线：全局引导 + 增量优先级规划

这个方案分成三层。

#### A. 全局引导层

先对地图做结构化分析，识别：

- highway / 主干道；
- bottleneck / 窄门；
- 停车位 / 等待位；
- 高需求服务区。

然后在路径代价中加入方向偏好与热点惩罚，让 agent 更倾向于：

- 沿主干道单向流动；
- 少做对向穿越；
- 少在服务区出口停滞。

这部分最接近：

- `Traffic Flow Optimisation for Lifelong Multi-Agent Path Finding`
- `Online Guidance Graph Optimization for Lifelong Multi-Agent Path Finding`
- `The Study of Highway for Lifelong Multi-Agent Path Finding`

#### B. 增量重规划层

每轮不重算所有 agent，而是先筛出受影响集合 `S`：

- 新接任务的 agent；
- 目标改变的 agent；
- 路径进入热点区的 agent；
- 被 executor 标记为长期卡顿的 agent。

对其余 agent：

- 保留已有 guide path；
- 或仅做非常轻的局部修复。

这一步是大规模可落地的关键。

#### C. 优先级规划层

对 `S` 中的 agent 计算优先级分数：

\[
Q_i = \eta_1 \cdot urgency_i + \eta_2 \cdot bottleneck_i + \eta_3 \cdot wait_i + \eta_4 \cdot taskDepth_i - \eta_5 \cdot detourRisk_i
\]

然后按 `Q_i` 顺序逐个规划，规划器可以使用：

- reservation table；
- space-time A*；
- SIPP 一类安全区间搜索；
- 或者你现有 planner 里的 guide path + local repair。

其本质类似 **prioritized planning / PBS 一类优先级搜索方法**：

- 顺序先定下来；
- 后规划者绕开前规划者的保留时空占用；
- 若某局部优先级导致代价过大，再在小子集内调整优先级。

### 14.3.4 beam search 适合放在哪

我不建议把 beam search 作为整个系统的主规划器，但它非常适合作为：

- **热点冲突子集的局部分支搜索器**；
- **优先级顺序候选器**；
- **瓶颈穿越顺序优化器**。

一种实用用法是：

#### A. beam 的节点表示

每个 beam node 可以表示：

- 一组冲突 agent 的优先级顺序；
- 或者一组局部候选路径前缀；
- 或者某 bottleneck 的通过序列。

#### B. beam 的评分函数

\[
F(node)= \alpha \cdot conflicts + \beta \cdot pathLength + \gamma \cdot hotspotPenalty + \delta \cdot delayRisk
\]

每一层只保留 top-`B` 个分支。

#### C. beam 的使用范围

beam 最适合：

- `|S|` 不大，但冲突结构比较复杂；
- 某个路口 / 窄门 / 交换区反复卡住；
- 希望在多个 passing order 之间快速比较。

它不太适合：

- 对全部 2000~5000 个 agent 同时展开。

因此 beam 更适合作为：

- **局部 conflict resolver**，而不是全局 solver。

### 14.3.5 为什么优先级路线比“全局最优搜索”更现实

因为这道题同时有：

- 大规模 agent；
- 在线补任务；
- delay；
- 有限规划预算；
- executor 会继续推进仿真。

在这种前提下，真正更现实的要求是：

- 每轮能稳定返回；
- 返回的路径对执行器友好；
- 允许后续继续修，而不是一口气求一个昂贵最优解。

这正是优先级增量路线的优势。

### 14.3.6 一个建议的整体规划 pipeline

如果让我设计，我会写成下面这个管线：

1. **根据新任务和执行反馈确定受影响集合 `S`**；
2. **更新 congestion / hotspot / highway 权重**；
3. **给 `S` 中 agent 计算优先级 `Q_i`**；
4. **按优先级做 reservation-based 增量规划**；
5. **对最难的 bottleneck 子集启用 bounded-width beam search**；
6. **输出多步计划，并给 executor 附带 bottleneck 顺序 / 等待点信息**；
7. **下一轮仅修复失效部分，而不是全部推倒重来**。

### 14.3.7 最终建议

对于路径规划，我最推荐的不是“纯 beam”也不是“纯全局最优”，而是：

- **主干方法**：全局引导 + 增量优先级规划；
- **局部增强**：在热点冲突子集上使用 beam search；
- **执行落地**：交给 delay-aware executor 去做每 tick 的最后放行。

### 14.4 三部分合起来，我最推荐的完整方案

如果把前面三部分拼起来，我认为当前题目最务实的一版完整方案是：

#### A. 调度层

- 初始化：**区域引导 + 局部匹配**；
- 动态阶段：**事件驱动增量匹配**；
- 成本函数里显式考虑：距离、任务链、区域负载、热点惩罚、切换惩罚。

#### B. 规划层

- 地图级别维护 highway / hotspot / waiting node；
- 每轮只对受影响集合做增量规划；
- 用优先级规划做主干；
- 对局部最难冲突用 beam search 做 bounded-width repair。

#### C. 执行层

- 用短窗口风险感知 TPG 做 `GO/STOP`；
- 引入等待年龄、瓶颈优先级和死锁破坏规则；
- 把长期失配 agent 回报给 planner 做下一轮修复。

这三层对应的关系非常清楚：

- **调度减少错误目标选择**；
- **规划减少结构性拥堵**；
- **执行减少 delay 导致的吞吐损失**。

### 15. 我的建议结论

如果让我给一个最务实的建议，我会这样排优先级：

- **第一优先级**：做更好的调度评分，把拥堵、区域负载和任务链成本纳入考虑；
- **第二优先级**：让 planner 少制造热点冲突，强调滚动规划和交通组织；
- **第三优先级**：让 executor 在 delay 下更少无谓 `STOP`，并具备基本死锁破坏能力。

一句话概括：

**这题最有希望的方向，不是追求单次最优路径，而是把“任务分配、交通组织、鲁棒执行”三件事协同起来，稳定做高吞吐。**
